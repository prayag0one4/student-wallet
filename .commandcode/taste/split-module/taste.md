# split-module
- For the Group Dashboard, use exactly two tabs: Overview (member balances list like WhatsApp chat list) and Expenses (unified financial timeline containing both expenses and payments in reverse chronological order). Payments use a visually distinct card style (different icon, badge, background, layout) from expense cards so users can immediately tell them apart. Do not show a single scroll view or a Settle Up card. Confidence: 0.70
- The FAB inside Group Dashboard opens a bottom sheet with three options: New Expense, New Payment (direct member-to-member payment), and New Person (add member inline). This replaces the separate Add Member screen and old Settle Up flow. Confidence: 0.70
- In the Overview tab, show each member with avatar, name, and net group balance (+₹X for receiving, -₹X for owing, ₹0 for settled). Green=receives, orange/red=owes, gray=settled. Do NOT show direct settlement relationships like "Rahul owes Prayag ₹250" — those only appear during the payment/settlement flow. Confidence: 0.70
- Treat direct payments between members as first-class transactions through the FAB "New Payment" option. These automatically reduce outstanding balances, preserve payment history, and recalculate optimized settlements. This replaces the separate SettlementScreen and Settle Up card mechanism. Confidence: 0.65
- The FAB bottom sheet serves ONLY as a launcher/menu (New Expense, New Payment, New Person). Selecting any option navigates to a dedicated full-screen creation flow — never open a second modal or nested bottom sheet from the FAB sheet. Full screens should use standard AppBar with back navigation. Confidence: 0.70
- Use `autofocus: false` on all text fields. Do not call `requestFocus()` inside initState(), build(), or post-frame callbacks. The keyboard should only appear after the user intentionally taps an input field. Confidence: 0.85
- Do not auto-select any member as the payer. The Paid By field must show a neutral placeholder (e.g., "Select Member") by default and require explicit user selection. The app must never assume who paid. Confidence: 0.85
- Allow single-participant expenses (minimum 1 participant). The payer is not required to be a participant in the expense. Confidence: 0.70
- The Overview tab displays ONLY member balances (avatar, name, net balance with +₹X/-₹X/₹0). Do NOT show settlement suggestions, debt relationships, or "Suggested Settlements" cards in the Overview tab. Settlement recommendations belong on a dedicated Settlement Suggestions screen accessed via a separate button. Confidence: 0.80
- Add a "Settlement" button in the Group Dashboard app bar or as a secondary FAB that navigates to a dedicated Settlement Suggestions screen. The screen shows optimized debt-simplified suggestions (minimal transactions) with payer avatar, receiver avatar, amount, and a checkmark button per row. Confidence: 0.75
- When the user taps the checkmark on a settlement suggestion, automatically create a Payment transaction, insert it into the Expenses timeline, update member balances, recalculate suggestions, and remove completed settlements — without requiring the user to manually open New Payment. Confidence: 0.70
- Both settlement-confirmed payments and manually created payments should appear identically in the Expenses timeline. Users should not be able to distinguish the origin of a payment transaction. Confidence: 0.65
